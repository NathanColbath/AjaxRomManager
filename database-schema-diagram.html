🔗 Entity Relationships

One-to-Many: Platform → ROMs (One platform can have many ROMs)
One-to-One: ROM → ROM Metadata (Each ROM has one metadata record)
One-to-Many: User → User Preferences (One user can have many preferences)
One-to-Many: User → System Logs (One user can generate many logs)
One-to-Many: Platform → Scan Jobs (One platform can have many scan jobs)
Many-to-Many: User ↔ ROMs (Users can favorite multiple ROMs, ROMs can be favorited by multiple users)

📝 Implementation Notes

  • Indexes: Add indexes on frequently queried fields like UserId, PlatformId, RomId, and Timestamp columns for optimal performance.
  • Soft Deletes: Consider implementing soft deletes for Users and ROMs to maintain data integrity and allow recovery.
  • File Integrity: The FileHash field enables duplicate detection and file integrity verification.
  • JSON Fields: ScreenshotPaths and Tags use JSON for flexible storage of arrays and complex data structures.
  • Audit Trail: System Settings include modification tracking for compliance and debugging purposes.
  • Scalability: Design supports horizontal scaling with proper foreign key relationships and normalized structure.
  • Security: Password hashing, encrypted settings, and role-based access control are built into the design.
  • Future Expansion: The potential tables section provides a roadmap for advanced features and community functionality.
